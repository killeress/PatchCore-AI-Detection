# Dust Filter 漏放問題分析

**日期**: 2026-03-31
**案例來源**: issue01.png
**狀態**: 分析完成，待選定方案實作

---

## 問題描述

一張影像中同時存在**兩個黑點**：
1. **表面灰塵** — 訊號較強、較明顯
2. **真實缺陷 (Defect)** — 訊號較弱

系統最終判定為 `DUST (Filtered as OK)`，**真實缺陷被漏放**。

### Debug 資訊 (來自 issue01.png)

- Score: **1.0000**
- 判定: **DUST (Filtered as OK)**
- COV: **0.818 >= 0.100**

---

## 根因分析

### 漏放流程

```
Heatmap 產生 → 兩個紅點都可見 (灰塵較亮、缺陷較暗)
        ↓
Top 0.2% 二值化 → 只剩灰塵那個點 (缺陷訊號不夠強，在這步被丟棄)
        ↓
OMIT 交叉驗證 → 確認是灰塵 → COV=0.818
        ↓
整張影像判定 DUST → Filtered as OK
        ↓
真實缺陷被一起放掉
```

### 核心問題

**問題出在二值化階段**：Top 0.2% 閾值只保留最亮的區域，當灰塵訊號強於真實缺陷時，缺陷在二值化時就已經消失，後續任何判定邏輯都無法救回。

### 閾值調整的矛盾

- Top% 調低（更嚴格）→ 弱訊號缺陷被切掉，只留灰塵
- Top% 調高（更寬鬆）→ 二值化一大片，失去定位精度，增加 false positive

這不是單純的閾值問題，而是架構問題。

---

## 改善方案評估

### 方案總覽

| # | 方法 | 是否在二值化前介入 | 本案例可行性 |
|---|------|-------------------|-------------|
| 1 | Connected component 分離 | 否 | ❌ 二值化時缺陷已丟失 |
| 2 | 多層閾值策略 | 是 | ⚠️ 可行但有 FP 風險 |
| 3 | Dust mask 後殘留檢查 | 否 | ❌ 二值化時缺陷已丟失 |
| 4 | OMIT 訊號相減 | 是 | ✅ 可行 |
| 5 | 高分異常保護 | 否 | ❌ 高分的是灰塵，方向相反 |
| 6 | Peak 數量守恆 (改良版) | 是 | ✅ 可行 |

### 可行方案詳細比較

#### 方法 4：OMIT 訊號相減

**流程**：
```
Heatmap (連續分數)          OMIT 影像
  ↓                           ↓
  ↓                     Top-Hat + Otsu → dust mask
  ↓                           ↓
  ↓                     轉為連續權重圖 (0~1)
  ↓                           ↓
  └──── Heatmap - 權重圖 ─────┘
              ↓
        殘差圖 (灰塵被壓低，缺陷保留)
              ↓
        Top% 二值化
              ↓
        殘差有異常 → NG / 無異常 → OK
```

**核心思路**：先減再切，不是先切再判。

| 優點 | 缺點 |
|------|------|
| 一步完成灰塵分離 | OMIT 與 heatmap 值域不同，normalization 是關鍵難點 |
| 弱訊號缺陷不被強訊號灰塵淹沒 | 減法係數調不好會過度壓制或不足 |

#### 方法 6 改良版：Peak detection on raw heatmap

**流程**：
```
Heatmap (連續分數)
  ↓
Local maxima detection (不經過 top% 二值化)
  ↓
找到 N 個 peak (例如: Peak A=0.95, Peak B=0.60)
  ↓
對每個 peak 獨立做 OMIT 交叉驗證
  ├── Peak A: 與 OMIT 重疊高 → 灰塵
  └── Peak B: 與 OMIT 無重疊 → 真實缺陷 → NG!
  ↓
任一 peak 判定為非灰塵且超過閾值 → NG
```

**核心思路**：不做全圖二值化，逐一找峰值、逐一判定。

| 優點 | 缺點 |
|------|------|
| 每個異常點獨立評估，不互相干擾 | 需設定 peak detection 參數 (prominence, min distance) |
| 邏輯清晰，每個 peak 有獨立判定報告，debug 友善 | 雜訊可能產生假 peak |
| 不需 normalization，直接用原始 heatmap 值 | 實作改動較大，流程從整張判定變為逐點判定 |

### 兩方案比較

| | 方法 4 (相減) | 方法 6 (逐 peak) |
|--|--------------|------------------|
| 二值化時機 | OMIT 相減之後 | 不做全圖二值化 |
| 評估單位 | 整張影像 | 每個 peak 獨立 |
| 弱訊號缺陷 | 減完後浮現 | 直接被找到 |
| 實作難度 | 中 (校準是關鍵) | 高 (流程重構) |
| Debug 可視性 | 中 (可看殘差圖) | 高 (每個 peak 有報告) |
| 誤判風險 | 減法係數不準會誤判 | 假 peak 會增加 FP |
